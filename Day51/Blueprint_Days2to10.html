Day 2 — Project Setup & Architecture Skeleton

🎯 Objective

Stand up the full-stack skeleton: frontend shell, backend server, and a working (test) call to the AI API end-to-end.

📖 What I'll learn

  • Structuring a frontend+backend project cleanly
  • Keeping API keys server-side via environment variables
  • Basic Express/Flask routing

🛠 Features to build

  • Landing/onboarding screen with resume paste textarea
  • Backend server with one test endpoint that calls the AI API and returns a response

📝 Step-by-step plan

  • Choose stack today: Node.js/Express (recommended, matches JS comfort) or Python/Flask
  • Create repo structure (see Files below)
  • Initialize backend project, install web framework + AI SDK + dotenv/cors
  • Create .env with AI API key; add .env to .gitignore
  • Build one route, e.g. POST /api/test, that sends a hardcoded prompt to the AI API and returns the text response
  • Build the frontend shell: header, hero, "Paste your resume" textarea, "Start Interview" button (disabled until text entered)
  • Wire a simple fetch() from frontend to /api/test to confirm connectivity

Day 3 — Claim Extraction Engine

🎯 Objective

Turn pasted resume text into a structured list of "defensible claims" the interview engine will use.

📖 What I'll learn

  • Prompt engineering for structured JSON output
  • Parsing/validating AI JSON responses safely

🛠 Features to build

  • POST /api/extract-claims backend endpoint
  • Frontend flow: paste → "Analyze Resume" → loading state → claims preview screen

📝 Step-by-step plan

  • Write a system prompt instructing the AI to act as a resume analyst and output ONLY JSON: array of claims, each with id, text, category (project/skill/internship/certification/academic/behavioral), priority (1–3)
  • Backend route sends resume text + system prompt to AI API, strips markdown fences, JSON.parse()s the result with try/catch fallback
  • Cap extraction to top 8 claims by priority (matches the 5–8 question session)
  • Frontend: on receiving claims, render them as a simple review list ("Here's what we'll ask you about") before starting the interview
  • Add a "Start Interview" button on this review screen that passes the claims array into the chat session (Day 4)

📂 Files/folders

backend/routes/extractClaims.js
backend/prompts/claimExtraction.js
frontend/claimsReview.js (or section in app.js)

🔗 Integrations

  • Same AI API as Day 2, new prompt/endpoint

🧪 Testing

  • Test with 3 different real resumes (short/long/technical-heavy)
  • Confirm JSON parsing doesn't crash on malformed AI output (add fallback retry)
  • Confirm claim count is capped at 8

🐞 Common issues

  • AI wraps JSON in ```json fences → strip fences before parsing
  • AI returns prose instead of JSON → strengthen system prompt with "Respond with ONLY valid JSON, no explanation"
  • Empty/garbage resume text → add minimum character length validation on frontend

✅ End-of-day checklist

  • ☐ Extraction endpoint returns clean structured claims
  • ☐ Claims review screen renders correctly
  • ☐ Tested with 3+ different resumes

Day 4 — Adaptive Interview Chat Engine

🎯 Objective

Build the core product experience: a live, adaptive chat-style AI interview covering 5–8 claims.

📖 What I'll learn

  • Maintaining multi-turn conversation context with an AI API
  • Designing an AI "persona" system prompt
  • Managing chat UI state

🛠 Features to build

  • Chat UI (message bubbles, input box, send button, typing indicator)
  • POST /api/interview-turn — takes conversation history + claims + current progress, returns next AI message
  • Session state tracking (question count, which claims covered)

📝 Step-by-step plan

  • Design the persona system prompt: "You are a skeptical but fair technical interviewer... reference the candidate's previous answer... escalate depth... never ask more than 8 questions total"
  • Backend endpoint accepts: full message history array + claims list + question count so far
  • Each backend call sends full conversation history to the AI so it has context (no need for the AI to "remember" — you resend history each turn)
  • Backend enforces hard cap: if question count ≥ 8, instruct AI to close the interview instead of asking another question
  • Frontend: on user send, append user message to chat, show typing indicator, call backend, append AI reply
  • Track session data in a JS object: {claims, messages, questionCount, startedAt}

📂 Files/folders

backend/routes/interviewTurn.js
backend/prompts/interviewerPersona.js
frontend/chat.js
frontend/chatUI.css

🔗 Integrations

  • Same AI API, conversational endpoint with history management

🧪 Testing

  • Run a full 8-question mock interview end-to-end
  • Confirm follow-ups reference previous answers (not generic/repeated)
  • Confirm session correctly stops at question 8

🐞 Common issues

  • AI keeps asking after 8 questions → enforce cap in backend logic, not just prompt instruction
  • Questions feel repetitive/generic → include the specific claim text + previous answer explicitly in each prompt, not just chat history
  • Long response latency → add a visible "AI is thinking..." indicator

✅ End-of-day checklist

  • ☐ Full chat UI functional
  • ☐ 8-question session runs end-to-end without crashing
  • ☐ Follow-up questions feel adaptive, not scripted

Day 5 — Scoring System

🎯 Objective

Score each user answer in the background on 4 axes, without disrupting the live chat experience.

📖 What I'll learn

  • Running a "silent" secondary AI call alongside a user-facing one
  • Designing simple weighted scoring rubrics

🛠 Features to build

  • POST /api/score-answer — scores one Q&A pair on Confidence, Technical Depth, Communication, Evidence Strength (0–100 each)
  • Frontend calls this silently after each user message (fire-and-forget, doesn't block chat)

📝 Step-by-step plan

  • Write a scoring system prompt: given the claim + question + user's answer, return ONLY JSON with 4 numeric scores + one-line reasoning per axis
  • Backend endpoint calls AI with this scoring prompt, parses JSON response with fallback defaults if parsing fails
  • Frontend: after each user answer is sent to the interview-turn endpoint, also fire a parallel call to score-answer (don't await before continuing chat)
  • Store all scores in session state array: {claimId, question, answer, scores:{confidence, depth, communication, evidence}, reasoning}
  • Compute simple weighted overall Readiness % as the average of all 4 axes across all answers

📂 Files/folders

backend/routes/scoreAnswer.js
backend/prompts/scoringRubric.js
frontend/scoring.js

🔗 Integrations

  • Same AI API, scoring-specific prompt

🧪 Testing

  • Confirm scores generate for every answer without blocking the chat UI
  • Test with a deliberately weak answer and a strong answer — confirm scores differ meaningfully
  • Confirm fallback scores apply if AI JSON parsing fails (never crash the session)

🐞 Common issues

  • Scoring calls slow down perceived chat speed → ensure it's truly async/non-blocking
  • Inconsistent score scales → constrain prompt to explicit 0–100 integer range
  • All answers scoring similarly → sharpen the rubric prompt with concrete criteria per axis

✅ End-of-day checklist

  • ☐ Every answer gets scored in background
  • ☐ Scores stored correctly per claim
  • ☐ Overall Readiness % calculates correctly

📸 Capture

Day 6 — Defense Report & Dashboard

🎯 Objective

Build the Defense Report screen: Readiness Dashboard + claim-by-claim analysis + AI Coaching Summary.

📖 What I'll learn

  • Turning scored data into meaningful visual dashboards
  • Prompting an AI to generate structured coaching narrative from structured data

🛠 Features to build

  • POST /api/generate-report — takes full session (claims, messages, scores) and returns coaching summary text
  • Defense Report UI: score dashboard (bars/rings), claim cards, coaching summary section

📝 Step-by-step plan

  • Compute dashboard numbers client-side from Day 5 scores: overall Readiness %, and averages for each of the 4 axes
  • Build simple CSS/SVG progress bars or ring charts for these 4 scores + overall %
  • For each claim, render a card: claim text, defense strength score, "what was convincing," "what was weak," "missing evidence," "suggested improvement" — pull the per-answer reasoning from Day 5, or make one more AI call to synthesize it cleanly
  • Backend generate-report endpoint: send the full session transcript + scores to AI, ask for JSON with strongestAreas, biggestRisks, storiesToPrepare, conceptsToRevise
  • Render this as the "AI Coaching Summary" section at the bottom of the report

Day 7 — Local History & Upload Support stretch: PDF/DOCX

🎯 Objective

Persist sessions locally so users can revisit past reports; add optional file upload if time allows.

📖 What I'll learn

  • Structuring localStorage as a lightweight database
  • Client-side PDF/DOCX text extraction (stretch)

🛠 Features to build

  • Save each completed session (profile snippet, claims, scores, report) to localStorage keyed by timestamp/ID
  • "Past Sessions" list screen to reopen any previous report
  • Stretch: PDF/DOCX upload with text extraction, falling back to manual paste on failure

📝 Step-by-step plan

  • Define a localStorage schema: dye_sessions = array of session objects
  • On report generation completion, push the session object into this array and save
  • Build a "History" screen: list of past sessions with date + overall Readiness % + click-to-reopen
  • Implement "replace active profile" — clears current claims/chat state but does not touch saved history
  • If time allows: add a file input accepting .pdf/.docx, use a client-side JS library to extract text, populate the paste textarea with extracted text, and let the user review/edit before analyzing
  • If extraction fails, show a friendly message: "Couldn't read this file — please paste your resume text instead" and keep the paste path fully functional

📂 Files/folders

frontend/history.js
frontend/storage.js
frontend/upload.js (stretch)

🔗 Integrations

  • Browser localStorage API
  • Stretch: a client-side PDF/DOCX text-extraction JS library

🧪 Testing

  • Complete 2–3 sessions and confirm all appear correctly in history
  • Reload the page and confirm history persists
  • If upload built: test with a real PDF and a corrupted/unsupported file to confirm fallback works

🐞 Common issues

  • localStorage size limits → store only essential session data, not full raw AI responses
  • JSON.parse errors on corrupted storage → wrap all localStorage reads in try/catch with empty-array fallback

✅ End-of-day checklist

  • ☐ Sessions persist across reloads
  • ☐ History screen functional
  • ☐ Upload stretch feature attempted only if core history is solid

Day 10 — Deployment & Documentation

🎯 Objective

Ship v1.0 — deploy backend + frontend publicly and finalize documentation for submission.

📖 What I'll learn

  • Deploying a Node/Flask backend to Render or Vercel
  • Managing environment variables in production
  • Writing clear project documentation

🛠 Features to build

  • No new features — deployment configuration only

📝 Step-by-step plan

  • Push final code to GitHub (if not already)
  • Deploy backend to Render (or Vercel serverless functions) — add the AI API key as an environment variable in the hosting dashboard, never in code
  • Deploy frontend (static) to Vercel/Netlify/GitHub Pages, pointing its API calls at the deployed backend URL
  • Update CORS settings on backend to allow the deployed frontend's domain
  • Do a full live smoke test on the deployed URL: paste → interview → report → history reload
  • Write final README.md: project description, setup instructions, live link, screenshots, tech stack, and "future scope" section

📂 Files/folders

README.md (finalized)
.env.example (documenting required env vars, no real keys)

🔗 Integrations

  • Render/Vercel hosting dashboards

🧪 Testing

  • Full live smoke test on public URL from a separate device/browser
  • Confirm no console errors referencing missing API keys or CORS failures

🐞 Common issues

  • CORS blocked in production → double check deployed frontend origin is explicitly whitelisted on backend
  • Env vars not picked up → confirm they're set in the hosting platform's dashboard, not just local .env
  • Cold start delays on free-tier hosting → mention this expected behavior in README

✅ End-of-day checklist

  • ☐ Frontend live and reachable
