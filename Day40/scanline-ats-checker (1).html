    How this was built

    The system prompt (the assistant's brain)

    Every scan sends the resume and job description to Claude with a system prompt that fixes its role tightly: act as an ATS-simulation layer, not a general career coach. It's instructed to return only raw JSON matching a strict schema (score, verdict, summary, matched/missing keyword arrays, prioritized fixes) — no prose, no markdown fences — so the interface can parse it deterministically every time.

    Edge cases are handled explicitly in the prompt rather than left to chance: if the pasted text isn't recognizable as a resume, or the job description is missing or too short, the model returns a distinct error field instead of guessing at a score. If either input contains instructions aimed at the model itself (a classic prompt-injection risk when parsing arbitrary pasted text), the prompt tells Claude to treat that text as inert content to analyze, never as commands to follow.

    Why the UI looks like this

    The visual language is borrowed from what an ATS actually does: optically scan a document. That's why the results panel has a scanning-beam loading state, a dial that reads like an instrument readout, and a monospace display face — it's meant to feel like machine-reading, not a chat window. Matched and missing keywords use an underline treatment rather than solid highlighter blocks, so the resume text stays fully readable while still making the gaps scannable at a glance.

    The dial and highlighted resume were prioritized over a long written report because the single most useful thing a student can walk away with is a clear signal (will this likely pass a filter?) plus the specific words to add — not a wall of advice.

    Extending it

    • Add a tool call: give Claude a web-search tool to pull real-time info about the target company's tech stack or common ATS platforms (Workday, Greenhouse, Taleo) and tailor keyword advice to that specific system's quirks.
    • Add memory: store previous scans per resume version in localStorage-free in-memory state during a session, so a user can compare score deltas after editing their resume, without needing a backend.
    • Multi-step flow: turn the single scan call into a two-step chain — first call extracts and ranks the JD's real requirements, second call scores the resume against that ranked list — for more consistent scoring across many resumes against the same JD (useful for the "recruiter batch-screening" variant of this tool).
    • File upload: add PDF/DOCX parsing client-side (e.g. pdf.js) so users can drop a file instead of pasting text.
