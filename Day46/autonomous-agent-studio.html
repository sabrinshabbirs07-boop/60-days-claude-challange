How the agents work together

Planner

Planner

Turns the topic and brief into a structured outline: angle, sections, and the takeaway the reader should leave with. Runs once, before drafting begins.

Executor

Executor

Writes the first full draft from the Planner's outline. Runs once; every subsequent draft comes from the Improver instead.

Evaluator

Evaluator

Scores the current draft 0–10 against a fixed rubric (clarity, engagement, structure, evidence, flow) and returns its reasoning. Runs every round.

Critic

Critic

Reads the draft and the Evaluator's score, then names the two or three specific changes that would move the score most. Runs every round.

Improver

Improver

Rewrites the full draft using the prior evaluation and critique as instructions. Its output becomes next round's draft. Runs every round the loop continues.

Final Reviewer

Final Reviewer

Runs once, after a stop condition fires. Gives the draft a last polish pass and writes a short editorial note on what changed across the run.

The stop-check is what actually governs information flow: after every Evaluator→Critic pass, the pipeline checks (in order) whether the score has plateaued for two rounds, whether it has crossed the target threshold, or whether the hard iteration cap has been hit as a safety fallback. Only if none of those fire does the Improver get called and the loop repeat — otherwise control branches to the Final Reviewer.

Architecture overview

Client-side only: every "agent" is a distinctly-prompted call to POST /v1/messages, orchestrated by a JavaScript while loop that threads a running history[] array forward as context. No server, no fixed schedule — the number of rounds is a runtime outcome of the stop-check, not a configured constant.

Extension ideas

  • Add a Safety Monitor agent that screens each draft for factual overreach or unsupported claims before it reaches the Evaluator.
  • Add a Memory Manager that compresses the history array into a short "lessons so far" memo once rounds exceed 4, so the Improver's context doesn't grow unbounded.
  • Let the Critic vote on which rubric dimension to prioritize next round, instead of always weighting all five equally.
  • Persist runs to localStorage so past pipelines can be compared side-by-side.
  • Swap the Executor/Improver for a multi-draft branch-and-select pattern: generate two drafts per round, let the Evaluator pick the stronger one to carry forward.

Further prompts to build on this

  • "Add a Safety Monitor agent that runs before the Evaluator and can force an early stop if it flags a factual problem."
  • "Turn the rubric into user-editable weighted sliders that change what the Evaluator's system prompt asks for."
  • "Add a second workflow preset — SEO article — with its own Planner rubric and stop condition, selectable before launch."
